# 小青蛙输出质量闸 v2

> Inspired by obra/superpowers verification-before-completion + taste-skill anti-slop + compound-engineering Recipe 规范。三段关卡：内容质量 → 验证证据 → 代码保全。

---

## 铁律：有证据才能说完成

```
没有当轮执行的验证输出 → 不能声称通过
```

**核心原则：证据先于断言，永远。**

**违反文字就是违反精神。**

### 闸门函数

```
声称任何状态之前：

1. 识别：什么命令可以证明这个声明？
2. 执行：运行完整的命令（重新执行，全新）
3. 读取：完整的输出，检查 exit code，数失败数
4. 验证：输出是否确认了声明？
   - 否 → 用证据报告实际状态
   - 是 → 带着证据说通过了
5. 只有到这一步才能发表断言

跳过任何一步 = 撒谎，不是验证
```

### 常见失败

| 声明 | 需要 | 不够 |
|------|------|------|
| 测试通过 | 测试命令输出：0 failure | 前一次的运行、"应该能过" |
| 代码正确 | 编译/构建结果：exit 0 | lint 通过、看起来对 |
| Bug 修复 | 重现原始症状→修复后通过 | 代码改了、假设修了 |
| 任务完成 | checklist 逐项确认 | "感觉做完了" |
| 质量检查 | 运行了 pre-flight，分数 >=7 | "我在脑子里过了一遍" |

### Red Flags — 停

- 用"应该"、"可能"、"看起来"
- 验证前就说"完美"、"好了"、"搞定了"
- 要 commit/push/PR 了还没验证
- 相信 subagent 的成功报告
- 只做了部分验证
- 累，想收工
- **任何没跑验证就暗示成功的措辞**

### Rationalization Table

| 借口 | 现实 |
|------|------|
| "应该能行了" | 执行验证。 |
| "我有信心" | 信心 ≠ 证据。 |
| "就这一次" | 没有例外。 |
| "subagent 说成功了" | 独立验证。 |
| "我累了" | 累不是借口。 |
| "只检查一部分就够了" | 部分验证证明不了什么。 |

## 2. HARD BANS — AI TELLS（中文）

以下模式禁止出现在最终输出。发现即重写，不是补丁。

### 2.A 结构与逻辑 Tell

| 模式 | 原因 | 修法 |
|------|------|------|
| `首先……其次……最后……` | LLM 默认列表结构 | 用自然过渡重组 |
| `总而言之` / `综上所述` | LLM 结尾拐杖 | 自然收尾，不用公式 |
| `值得注意的是` | 被滥用的 AI 路标 | 直接说要点 |
| `从某种意义上说` | 无目的的委婉 | 下结论或删掉 |
| `不可否认的是` | 虚假平衡 | 说观点或放弃 |
| `在当今时代` / `随着……的发展` | AI 文章开头惯用语 | 用实质内容开头 |
| `随着人工智能的迅猛发展` | AI 万能开头 | 直接进入主题 |

### 2.B 词汇 Tell

| 模式 | 原因 | 修法 |
|------|------|------|
| `赋能` | #1 中文 AI 流行词 | 用具体动词：帮助/支持/让……能 |
| `关键作用` / `重要作用` | 填充名词短语 | 展示，不要贴标签 |
| `不仅……而且……` | LLM 并行结构瘾 | 拆两句或砍一半 |
| `提供有力支撑` | 官僚 AI 话 | 说它实际做什么 |
| `深入探讨` 但内容浅显 | 虚假深度标签 | 匹配深度和声明 |
| `众所周知` | 居高临下 | 直接陈述事实 |

### 2.C 格式 Tell

| 模式 | 原因 | 修法 |
|------|------|------|
| `——` 破折号分隔 | 视觉 AI 签名 | 逗号、句号或冒号 |
| 随机**粗体**强调 | AI 过度加粗 | 关键术语最多每段1个 |
| 全是列表 | 偷懒结构 | 正文段落、编号步骤或表格 |
| `（注：……）` 括号说明 | AI 对清晰度没信心 | 融入句子或删掉 |

### 2.D 语气 Tell

| 模式 | 原因 | 修法 |
|------|------|------|
| 过度解释显而易见的事 | AI 不知道读者知道什么 | 信任读者智力 |
| 虚假平衡"一方面……另一方面……" | AI 回避立场 | 站边或呈现为选项 |
| 过多`我们` | AI 伪社区感 | 少用，用具体主语 |
| `让我们` 邀请 | AI 导游腔 | 直接陈述 |

## 3. PRE-FLIGHT CHECKLIST

交付前必须跑。Score 0-10。低于 7 分不发。

### 3.A 完整性 (0-3)
- [ ] 无 `TODO`、`FIXME`、`[...]`、`（待补充）` 占位符
- [ ] 无截断段落（检查最后3段完整性）
- [ ] 所有代码块闭合、链接格式有效
- [ ] 所有图片有 alt 文本

### 3.B 一致性 (0-2)
- [ ] 统一语气（无中间变调）
- [ ] 术语一致（同一个词同一翻译）
- [ ] 格式一致（标题级别、列表样式、代码围栏）

### 3.C AI-Tell 扫描 (0-3)
- [ ] 零个 SECTION 2 禁用模式
- [ ] 无连续两段同样的开头方式
- [ ] 无连续 3+ 句同样开头

### 3.D 可读性 (0-2)
- [ ] 段落长度有变化（短+中混排，无墙式文本）
- [ ] 英语输出：无理由不超过 40 词一句
- [ ] 标题具体可扫描

## 4. RECIPE 输出规范（v2 新增）

当输出的"形状"需要严格一致时（如项目报告、评估摘要、设计文档），用 **Recipe** 模板替代散文描述。

### Recipe 的结构

```
输出必须有且只有以下部件，按此顺序：

1. [判决/结论] — 一句话，可能是判断、分数、状态
2. [关键证据] — 支持判决的核心事实，不超过 3 条
3. [理由/分析] — 为什么这个判决成立
4. [下一步/建议] — 如果有的话
```

### Recipe vs 禁令 的选择

| 场景 | 用 | 示例 |
|------|-----|------|
| 输出形状有问题 | Recipe | "输出必须是：结论→证据→建议" |
| Agent 知道规则但违反 | 禁令+Red Flags | "违反文字就是违反精神" |
| 容易遗漏关键内容 | 结构模板 | 在模板里加 `## 风险（必填）` |

### Recipe 示例

```markdown
# ❌ 坏 — 形状不明确的输出
我检查了代码。init 函数看起来没问题。不过我觉得 init 的顺序可能需要调整。建议先做 A 再做 B。

# ✅ 好 — 用 Recipe 规范形状
## 评估结果
**状态:** ✅ 通过
**分数:** 8/10

**证据:** 
- init 函数执行成功，exit code=0
- 所有 34 个测试通过，0 失败

**分析:**
init 顺序正确，A 在 B 之前执行符合设计规范。

**建议:**
无。可以合入。
```

## 5. 条件谓语指令（v2 新增）

当行为的正确方式取决于上下文条件时，用条件谓语代替无条件规则。

### 错误方式：无条件规则 + 豁免条款

```markdown
# ❌ 坏 — 无条件规则 + 豁免条款
永远不要复述需求。除非需求很复杂，需要上下文。

# 为什么坏：
agent 在"这个需求很复杂"上找到豁免，永远会复述。
```

### 正确方式：条件谓语

```markdown
# ✅ 好 — 条件谓语
输出的格式取决于当前上下文：
- 如果需求 brief 已在对话中且你引用它：不要重复，直接说"按需求描述"。
- 如果需求 brief 不在对话中：用一句话总结上下文。
```

### 条件谓语原则

1. **条件必须基于可观察的谓词**（"需求 brief 在不在"）而不是主观判断（"复杂不复杂"）
2. **每个条件有明确的行为**，不留灰色地带
3. **条件全覆盖** — 加一个 else/默认分支兜底

---

## 6. SECTION-LEVEL AUDIT

每段拷问：
1. **值得存在吗？** — 能一句话概括合并到别的段？合并或砍。
2. **重复了吗？** — 重复 = 合并或砍。
3. **是不是该停了？** — 最后一段是最强出口，不是语尽。

---

## 7. THE CLEAN-PASS 规则

所有修正后，再读一次全文。没问题就发。发现问题就从那里往前修（不是从头）。

---

## 8. DISCLOSURE（输出质量受限时）

如果质量受限于输入质量（如原文乱码），在末尾加一行：
> ⚠️ 部分内容受限于原文质量，可能存在不够精确之处。

只在真正受限时使用，不躺平。
